Topic 1: Center for Disease Control and Prevention (CDC) Requirements
Tiered World Studios understands that the objective of the Hospital Bed Emergency
Data System (HBEDS) program is to establish a statewide operational platform. The
platform provides accurate, timely, and actionable visibility into staffed bed availability by
patient care type and level of service across hospitals and alternative care sites.
Assembly Bill 177 authorizes CDPH to establish and operate this statewide healthcare
bed capacity system. It requires hospitals, emergency departments, and behavioral
health facilities to report bed availability by level of care, unit, and facility.
This mandate reflects the critical role of real-time capacity visibility in patient outcomes
and emergency response. As healthcare systems approach capacity thresholds, access
to timely and accurate bed availability data becomes essential for patient routing,
resource allocation, and coordinated response.
HBEDS operationalizes this requirement as a continuous, statewide capability,
delivering real-time data across California‚Äôs healthcare system.

1.2 EHR/EMR Integration Approach
A core requirement of HBEDS is seamless integration with hospital Electronic Health
Record (EHR) and Electronic Medical Record (EMR) systems to collect staffed bed
availability data by bed type. The system is designed to receive this data through
one-way, outbound transmissions from hospital systems and does not collect any
Protected Health Information (PHI).
MeshMill meets this requirement through an interoperability framework designed for
compatibility with widely adopted healthcare data standards. As a result, the platform
integrates with all EHR/EMR systems used across California, including Epic, Oracle
Cerner, Meditech, and legacy platforms.
Integration is rapid through a configuration-driven approach. Hospitals connect using
existing standard interfaces without custom development. The scope of data transmitted
to MeshMill is limited to staffed bed availability by configuring and mapping only
required fields to the CDPH schema at the point of ingestion. Each hospital‚Äôs data
structures are mapped through a one-time configuration process, enabling efficient
onboarding at scale.
MeshMill‚Äôs core architecture is designed to normalize heterogeneous data across
disparate systems and is applied in this implementation using standards such as FHIR
R4, HL7 v2, and SFTP.
The platform supports multiple integration methods aligned with hospital capabilities:
‚óè‚Äã FHIR R4 APIs for modern, standards-based interoperability
‚óè‚Äã HL7 v2 messaging for compatibility with legacy systems
‚óè‚Äã Secure File Transfer Protocol (SFTP) for batch data exchange
‚óè‚Äã RESTful APIs for flexible, real-time integrations
This multi-modal integration approach ensures compatibility with both modern and
legacy systems.
Data Scope and Data Controls‚Äã
MeshMill is designed to collect only staffed bed availability data by bed type, limited to non-PHI
operational capacity metrics as defined by CDPH.
Data is received through one-way, outbound transmissions from hospital systems. MeshMill
does not access hospital EHR systems and does not retrieve or query patient records.
The platform explicitly defines which data elements are accepted and how they map to the
CDPH schema. Only required fields are ingested. All other data elements are excluded at the
point of ingestion.

The following example illustrates how MeshMill integrates real-time data from diverse
hospital systems into a unified statewide view for reporting and operational use.
Data Flow
A hospital using Epic securely transmits staffed bed availability and related data
elements through FHIR APIs. Another hospital may transmit similar data using HL7 v2
interfaces. Additional facilities may provide data through SFTP-based file exchanges.
These heterogeneous data inputs arrive in different formats, structures, and naming
conventions. MeshMill receives and ingests concurrent data streams from multiple
facilities for downstream processing.
Data Normalization
MeshMill normalizes heterogeneous EHR/EMR data into a consistent statewide
structure through a canonical data mapping architecture. Each hospital‚Äôs internal data
structures are translated into a standardized schema through a one-time configuration
process, establishing a consistent data foundation without ongoing customization.
At the core of this process is the M.A.R.I.O. Interoperability Engine. It functions as a
real-time data translation layer, resolving differences in formats, naming conventions,
and data structures across systems such as Epic, Oracle Cerner, MEDITECH, and
others.

Data Standardization and Scalability
MeshMill applies standardized definitions to ensure consistency across all participating
facilities. Bed types, capacity metrics, and reporting structures are aligned to a common
statewide model. Equivalent data elements are interpreted consistently, regardless of
how they are represented at the source system. Variations in terminology and reporting
are resolved to a uniform definition aligned with CDPH requirements.
This standardized model enables accurate aggregation and comparison across
hospitals and regions. It supports consistent reporting and a clear statewide view of
capacity. The platform applies these standards through a configuration-driven approach.
New hospitals are onboarded using the same model, enabling scalable deployment
across hundreds of facilities without additional customization.
Data Validation and Quality Assurance
MeshMill applies data validation processes to ensure the accuracy, consistency, and
reliability of all collected data. Validation occurs in real time and includes field-level
checks to confirm required data elements are present, correctly formatted, and within
expected ranges. MeshMill also applies logical validation rules to ensure internal
consistency. For example, available bed counts cannot exceed total capacity, and
related data elements must align.
The platform detects anomalies such as unexpected spikes, drops, or inconsistencies in
reported data and flags them for review. When issues are identified, MeshMill initiates
automated error handling, including record flagging, alert generation, and defined
resolution workflows. All validation activities are logged and auditable, providing full
traceability of data submissions. Continuous monitoring ensures rapid detection of data
interruptions or integration failures, minimizing impact to data availability and reporting
continuity.
1.4 CDC NHSN Reporting and Compliance
MeshMill will automate data formatting and submission to the CDC National Healthcare
Safety Network (NHSN). Data validation is performed prior to submission to ensure
accuracy and completeness. Required bed capacity data is transmitted at least twice
daily in alignment with CDC NHSN requirements, with support for more frequent
submissions as requirements evolve.
All submissions are fully auditable, with comprehensive logging of transmission activity,
including timestamps, record counts, and acknowledgment tracking. This provides full
transparency and traceability.

1.7 Security and Authentication
All data transmission uses HTTPS with TLS 1.3 encryption. FHIR API connections use
OAuth 2.0 and OpenID Connect (OIDC) for authentication.
Dashboard access is secured through Single Sign-On (SSO) using SAML 2.0 and
OIDC/OAuth2. Multi-Factor Authentication (MFA) is enforced for all privileged access,
sensitive data access, and official communication accounts. Role-based access control
(RBAC) restricts users to only the data and functionality required for their role.
The platform operates on a one-way, outbound data ingestion model and does not
access hospital EHR systems. No Protected Health Information (PHI) is transmitted,
processed, or stored. Hospitals control all data transmission through secure interfaces.
The platform is hosted on AWS, which provides secure, HIPAA-eligible cloud
infrastructure. Data is encrypted at rest, and audit logging is enabled across the
platform. A Business Associate Agreement (BAA) is in place with AWS.
Security is enforced at both the infrastructure and application levels. Authentication,
access control, session management, MFA enforcement, and audit logging are
managed within the application to ensure that security does not rely solely on the
underlying cloud infrastructure.

4.3 Automated Bed Reporting After Enrollment
MeshMill automates bed reporting after hospital enrollment and validation. The system
collects, processes, and transmits data continuously without manual input from hospital
staff.
Automated reporting delivers consistent, timely, and reliable data submission. Hospitals
maintain reporting compliance without additional operational burden.
Monitoring tools track data delivery and system performance. Support teams respond to
disruptions to maintain continuous reporting.
4.4 Transition to Steady-State Operations
The program transitions hospitals from onboarding to steady-state operations through
defined checkpoints. Teams validate data accuracy, confirm reporting performance, and
verify system readiness.
Once validated, hospitals operate in automated reporting mode with ongoing monitoring
and support. CDPH maintains continuous visibility into statewide data.
